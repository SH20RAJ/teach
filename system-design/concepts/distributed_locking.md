# System Design: Distributed Locking (Redlock & Fencing)
> **Format:** Concept-First Whiteboard Explanation
> **Topic:** Concurrency Control in Distributed Systems

---

## 1. First Understand the Problem

Imagine you run an online ticket platform.
There is **only 1 ticket remaining** for a highly popular VIP seat.
Two users, **User A** and **User B**, click "Buy Now" at the exact same millisecond.

Your application runs on three separate server nodes: **App Server 1**, **App Server 2**, and **App Server 3** behind a load balancer.

```text
User A ──► [ App Server 1 ] ─── (Tries to book Seat 10) ───┐
                                                          ├──► [ Database ] (Seat 10)
User B ──► [ App Server 2 ] ─── (Tries to book Seat 10) ───┘
```

### The Race Condition Sequence:
```text
Server 1 reads database: "Seat 10 status: Available"
Server 2 reads database: "Seat 10 status: Available"

Server 1 calls Payment Gateway: Charges User A.
Server 2 calls Payment Gateway: Charges User B.

Server 1 writes database: "Seat 10 booked by User A"
Server 2 writes database: "Seat 10 booked by User B" (Overwrites Server 1!)
```
This is a **Double Booking Disaster!** Both users were charged, but only one can sit in the seat.

In a single-threaded app, we could use a local programming language lock (like a Java `synchronized` block or a Go Mutex).
But our servers are on **different physical machines**. A Go Mutex on Server 1 has no idea that Server 2 is running.

We need a central lock that is shared across all servers.
That lock is a **Distributed Lock**.

## 2. Imagine the Office Whiteboard Marker Analogy

Imagine a busy office with one shared meeting room.
To prevent two teams from entering at the same time, the manager places a single **special red marker** on the main office table.

### The Rules:
1. If a team wants to use the meeting room, they must walk to the table and grab the Red Marker.
2. If they successfully grab it, they enter the room.
3. If another team wants to enter, they check the table. The marker is missing! They must wait.
4. When the first team finishes, they return the marker to the table (Release Lock).

```text
Team A ──► [ Grab Marker (Success) ] ──► Enter Meeting Room
Team B ──► [ Try Grab Marker (Failed) ] ──► Must wait...
```

### Mapping to System Design:
- **Meeting Room** = **Shared Resource** (the Seat 10 database row).
- **Red Marker** = **Lock Entry** in a fast centralized key-value store (like Redis or ZooKeeper).
- **Table** = **Redis Cluster**.

## 3. How to Implement a Safe Lock in Redis: Step-by-Step

Let's look at the implementation steps on the whiteboard.

### Step 1: Set the Lock Key
To acquire the lock, the app server runs a Redis command:
`SET seat_10_lock unique_uuid NX PX 10000`

- `NX`: "Set if Not Exists". If the key already exists, this command fails (the marker is taken!).
- `unique_uuid`: A random ID generated by the app server so it can identify that it owns this lock.
- `PX 10000`: **Lease Time** (Expiration). If the app server crashes mid-process, the lock will automatically delete itself in 10 seconds (10,000ms), preventing a permanent deadlock!

### Step 2: Release the Lock Safely
When the work is done, the server deletes the key.
But wait! What if Server 1 took 12 seconds to do its work (longer than the 10-second lease)?
1. Server 1's lock expired.
2. Server 2 acquired the lock because the key disappeared.
3. Server 1 finishes and calls `DEL seat_10_lock`.
4. Server 1 just deleted Server 2's lock!

**Solution:** The release must use a Lua Script that checks: *"Delete this key ONLY if the value inside matches my unique UUID."*

## 4. The GC Pause Crisis & Fencing Tokens

Even with Redis, a major system failure can occur due to **Stop-the-World Garbage Collection (GC) pauses**.

Let's watch this timeline trace:
```text
Time 0: Server 1 acquires lock with a 10-second lease.
Time 1: Server 1's JVM enters a Stop-the-World Garbage Collection pause (freezes completely!).
Time 11: The lock lease expires in Redis.
Time 12: Server 2 acquires the lock and starts writing to the Database.
Time 13: Server 1's GC pause ends. Server 1 wakes up, thinks it still holds the lock!
Time 14: Server 1 writes its data to the Database, overwriting Server 2!
```
```text
Lock Owner: [ Server 2 ]
Database:   [ Overwritten by zombie Server 1! ]
```

### The Solution: Fencing Tokens (Bookmarks)
Every time a lock is acquired, the lock manager returns a **fencing token** (a number that increments every time, e.g. 101, 102, 103).

When writing to the database, the server must pass its token:
```text
Server 2 writes with Token = 102. Database stores: Last Token = 102.
Zombie Server 1 wakes up and tries to write with Token = 101.
Database checks: 101 is LESS than 102!
Database rejects the write!
```

## 5. Common Confusion

### Confusion: Can we just use a database table with `status = 'locked'` as a distributed lock?
*   **Correction:** Yes, you can! This is called a database-backed lock. However, databases store data on disk and have high transaction overhead. For high-scale systems, using disk-backed databases for locking creates a severe bottleneck. centralized, in-memory systems (like Redis or ZooKeeper) are preferred for sub-millisecond response times.

---

## 6. Interview-Safe Definitions

1. **Distributed Lock:** A mechanism used to coordinate access to a shared resource across multiple independent execution environments (servers).
2. **Lease Time:** The duration of time after which a distributed lock is automatically released by the lock provider if not manually renewed or released.
3. **Fencing Token:** An auto-incrementing number returned by a lock service used to validate write authorization and reject out-of-order writes from frozen servers.
